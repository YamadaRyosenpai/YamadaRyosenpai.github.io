【DAY2】从零手写：基于 C++、Docker 和 MySQL 的高并发公网聊天室 & Python 数据采集

项目背景

作为一个计算机科班学生，我不满足于仅刷算法题。为了深入理解后端底层架构，我决定跳出舒适区，不使用现成的高级 Web 框架，而是利用 C++ 和 Linux 系统调用，从零手写一个支持多用户在线、数据持久化存储的聊天系统，并将其容器化部署到公网上。

同时，为了丰富系统的数据源，我额外开发了一个 Python 爬虫，通过数据清洗后存入同一个 MySQL 数据库，实现了多语言服务的联动。

技术栈一览

  • 核心服务: C++ (C++11/17 标准), Linux BSD Socket, STL Thread

  • 数据采集: Python 3.11 (Requests, BeautifulSoup, Pandas, PyMySQL)

  • 数据库: MySQL 8.0 (使用 C API libmysqlclient 及 Python pymysql)

  • 容器化: Docker & Docker Compose

  • 内网穿透: Pinggy (实现公网访问)

  • 开发环境: WSL 2 (Ubuntu 22.04) + VS Code

第一阶段：C++ 核心通信功能的实现

1. 建立 TCP 连接

抛弃了高级语言的 HTTP 库，我直接使用 <sys/socket.h> 进行开发，深入理解 TCP 三次握手在代码层面的体现。

  • 核心流程: socket() -> bind() -> listen() -> accept()。

  • 多线程模型: 采用 Thread-per-Client 模型。每当有一个新用户 accept 进来，就开辟一个新的 std::thread 专门处理该用户的消息，实现多人同时在线聊天。

2. 解决“楼梯式换行”问题

遇到的坑: 在使用 Windows Telnet 测试时，发现回显的消息呈“楼梯状”排列。 原因分析: Linux 默认换行是 \n (Line Feed)，而 Windows 终端需要 \r\n (Carriage Return + Line Feed) 才能回归行首。 解决方案: 优化了服务端的消息发送逻辑，手动补全 \r\n，并增加了行缓冲处理，确保收到完整的一句话后再进行广播。

第二阶段：数据持久化与业务逻辑

为了让聊天室具备实用性，我引入了 MySQL 数据库来存储用户信息和聊天记录。

1. 数据库设计

在 MySQL 中设计了两张核心表：

  • users: 存储 username 和 password，用于注册登录。

  • messages: 存储聊天记录，包含发送者、内容和时间戳。

2. 业务指令解析

我在 C++ 服务端实现了一个简单的指令解析器，支持以下命令：

  • #reg <user> <pass>: 注册账号。

  • #login <user> <pass>: 登录并获取历史消息。

  • #logout: 退出登录并广播通知。

3. 亮点功能：历史消息回放

当用户登录成功后，服务端会立即查询 messages 表中最近的 5 条记录发送给客户端，实现了类似微信的“漫游消息”体验，让新加入的用户也能跟上话题。

番外篇：Python 爬虫与环境踩坑

为了练习多语言协作，我编写了一个 Python 爬虫，抓取书籍数据并存入 MySQL。在这个过程中，我深刻体会到了“环境配置”的重要性。

1. 那个让我崩溃的 Python 3.13

遇到的坑: 起初我使用了最新的 Python 3.13，结果在安装 lxml 等依赖库时频频报错。原因是这些库的预编译二进制包（Wheel）尚未适配最新的 Python 版本。 解决方案: 果断卸载，降级到 Python 3.11（工业界最稳定的版本之一）。降级后，所有库瞬间安装成功。 感悟: 在工程开发中，稳定压倒一切，盲目追新会带来不必要的成本。

2. 爬虫架构

使用了经典的 Python 数据采集栈：

  • Requests: 模拟浏览器发送 HTTP 请求。

  • BeautifulSoup: 解析 HTML DOM 树，提取书籍信息。

  • Pandas: 进行数据清洗（去除货币符号、类型转换）和格式化，体验了比 C++ 方便得多的数据处理能力。

  • PyMySQL: 连接 WSL 中的 MySQL 数据库，将爬取到的书籍数据批量入库。

3. 极其隐蔽的 [WinError 10061]

在 Windows 运行 Python 连接 WSL 里的 MySQL 时，遇到了“目标计算机积极拒绝”的错误。 排查过程:

  1. 检查服务：MySQL 服务确实启动了。

  2. 检查配置：发现 bind-address 默认为 127.0.0.1，导致 MySQL 只监听 Linux 内部请求。

  3. 最终解决:

    • 修改 MySQL 配置 bind-address = 0.0.0.0。

    • 授权 Root 用户远程登录权限 (host = '%')。

    • 使用 WSL 的真实 IP（172.x.x.x）进行直连，成功打通 Windows 到 Linux 数据库的链路。

第三阶段：Docker 容器化改造

为了解决“在我的电脑上能跑，在别人电脑上跑不起来”的环境依赖问题（特别是 libmysqlclient 的版本差异），我决定引入 Docker。

1. 编写 Dockerfile

基于 ubuntu:22.04 镜像，自动安装 g++ 和 mysql-dev 依赖，并在构建时自动编译 C++ 代码。这确保了编译环境的一致性。

2. Docker Compose 微服务编排

这是整个架构的高光时刻。我不再手动在系统中安装 MySQL，而是通过 docker-compose.yml 编排了两个服务：

  • mysql-db: 运行官方 MySQL 8.0 镜像。

  • chat-server: 运行我编写的 C++ 程序。

遇到的坑: 在容器内连接数据库时，写 127.0.0.1 连接失败。 解决: 理解了 Docker 网络隔离机制后，明白容器内的 localhost 指的是容器自己。将连接地址改为服务名 db，利用 Docker 内部 DNS 成功连通。

第四阶段：走向公网

为了让朋友也能通过手机连接我的聊天室，我使用了内网穿透技术。

Pinggy 实战

放弃了配置复杂的 FRP 和需要绑卡认证的 Ngrok，我选择了 Pinggy。

  • 优势: 无需下载客户端，利用 SSH 协议直接穿透。

  • 命令: ssh -p 443 -R0:localhost:8888 tcp@a.pinggy.io

  • 成果: 成功实现了 手机 (4G网络) -> 公网 -> Pinggy 隧道 -> 本地 Docker -> C++ 服务器 的全链路贯通。手机端使用 TCP Client APP 即可远程注册登录。

总结与收获

通过这个项目，我不仅掌握了 C++ 网络编程的细节，更重要的是建立起了**“全栈工程化”**的思维。从底层的 Socket 字节流处理，到数据库的 SQL 交互，再到 Docker 的容器编排，每一环都亲手打通。

接下来的计划：

  1. 引入 JSON 库，替代纯文本协议，支持更复杂的数据交互。

  2. 引入 Redis 缓存，优化数据库查询性能。

  3. 研究 IO 多路复用 (Epoll)，将多线程模型升级为 Reactor 模型，挑战更高并发。
